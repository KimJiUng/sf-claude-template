# CLAUDE.md

이 문서는 이 저장소에서 Claude가 따라야 할 작업 지침의 기준 문서입니다.
Codex 사용 시의 대응 문서는 `AGENTS.md`입니다. 운영 규칙을 바꿀 때는 두 파일을 함께 갱신합니다.

## 우선 원칙

- 결과, 성공 기준, 제약, 출력 형식, 중단 조건을 먼저 파악하고 작업합니다.
- 사용자 요구사항의 누락 정보가 결과를 크게 바꾸거나 보안/권한/데이터/배포 리스크를 만들면 **구현 전에 좁게 질문**합니다.
- 명확하고 되돌릴 수 있는 저위험 작업은 합리적으로 가정하고 진행하되, 가정과 남은 확인 사항을 보고합니다.
- 작업은 역할 기반 오케스트레이션을 따릅니다.
- 설계/요구사항/운영 산출물은 저장소 내 적절한 문서(`docs/**`, `context/**`)로 남깁니다.
- Salesforce 소스 또는 Org 영향 작업 전에는 영향 파일의 로컬 백업과 Org 시작 snapshot을 생성합니다.
- Org 반영은 사용자가 명시적으로 승인하기 전까지 **절대 실행하지 않습니다.**
- Org 반영 전에는 Deploy Gate를 **반드시** 통과해야 합니다.
- **`context/`, `logs/`, 관련 운영 문서의 제목과 서술형 본문은 기본적으로 한글로 작성합니다.**
- **코드 식별자, 경로, 명령어처럼 원문 유지가 필요한 기술 식별자는 그대로 둘 수 있습니다.**
- **`context/` 폴더는 영속 컨텍스트(작업 메모리)입니다. 반드시 아래 규칙을 따릅니다.**
- **`logs/` 폴더는 실패 기록 보관소입니다. 기본 작업에서는 읽지 않습니다.**
- Claude는 기존 사용자 변경을 되돌리지 않고, 필요한 변경만 좁게 적용합니다.
- 변경 추적은 `backups/` 작업 snapshot과 문서 기록을 기준으로 합니다.
- 반복되는 문제나 사용자의 반복 요청은 재귀 개선 루프에 따라 관련 Markdown 문서에 반영합니다.

## 작업 출력 기준

- 시작 전에는 목표, 성공 기준, 제약, 가정, 영향 파일, 검증 방법을 짧게 정리합니다.
- 진행 중 중단 조건(사용자 승인 필요, 같은 영역 충돌, Deploy Gate 위반, 위험한 누락 정보)이 발견되면 작업을 멈추고 보고합니다.
- 완료 보고에는 변경 파일, 작업 요약, 검증 결과, 기술부채 후보, Org 반영 여부를 포함합니다.

## 하네스 방식 — 결과 중심 요구사항 확인

> Claude는 요구사항을 목표, 성공 기준, 제약, 출력, 중단 조건으로 정리한 뒤 작업합니다.
> 누락 정보가 결과나 리스크를 실질적으로 바꿀 때만 구현을 시작하기 전에 사용자에게 질문합니다.
> 기술적 판단(구현 방식, 아키텍처 선택)은 Claude가 자율적으로 수행합니다.

### 질문이 필요한 상황

아래 항목이 비어 있고, 합리적 가정으로 진행하면 결과가 달라지거나 의미 있는 리스크가 생기는 경우 질문합니다.

1. **비즈니스 목적이 불명확** — 무엇을 해결하려는 건지 모호할 때
2. **대상 오브젝트/필드가 특정되지 않음** — "고객 정보를 관리해줘" 같은 추상적 요청
3. **UI 요구사항 부재** — 화면이 필요한데 레이아웃/동작 기준이 없을 때
4. **비즈니스 규칙 누락** — 조건/예외/권한 범위가 정의되지 않았을 때
5. **테스트 기준 부재** — 완료 조건(인수 기준)을 판단할 수 없을 때

### 질문 프로토콜

```text
1. 요구사항을 받으면 위 5개 항목을 체크합니다.
2. 누락 항목이 결과/보안/권한/데이터/배포에 영향을 주면 가장 작은 질문만 제시합니다.
3. 저위험/가역 작업에서 합리적 가정이 가능하면 진행하고 가정과 확인 필요 항목을 보고합니다.
4. 사용자가 답변하거나 가정이 정리되면 context/open_tasks.md에 반영합니다.
5. 아래 조건에 해당하는 항목이 있으면 기술부채로 선등록합니다.
```

### 불확실 요구사항 → 기술부채 선등록

요구사항 분석 중 아래에 해당하는 항목이 발견되면 **구현 전에** `docs/technical-debt/register.md`에 `pending-confirm` 유형으로 등록합니다:

1. **고객사 확인 대기** — 고객사/담당자 컨펌이 필요한 비즈니스 규칙이나 정책
2. **변경 가능성 높음** — 사용자가 "일단 이렇게", "나중에 바꿀 수도" 등으로 표현한 항목
3. **가정 기반 구현** — 정보 부족으로 Claude가 명시적 가정을 두고 구현한 부분
4. **외부 의존성 미확정** — 연동 시스템/API 스펙이 확정되지 않은 경우

등록 형식:

```markdown
| TD-xxx | review-needed | pending-confirm | 항목 제목 | 영향 | `파일:라인` | YYYY-MM-DD | 검토 대기 |
```

구현 중 발견된 후보는 `review-needed` 상태로 누적하고, 사용자가 검토 후 `accepted`, `resolved`, `skipped` 중 하나로 정리합니다.

### 질문 없이 진행해도 되는 경우

- 명확한 버그 수정 요청 (재현 경로 + 기대 동작이 있음)
- 단순 리팩토링/코드 정리 지시
- 문서/설정처럼 되돌릴 수 있고 외부 영향이 없는 저위험 작업
- 사용자가 "판단에 맡긴다"고 명시적으로 위임한 경우

## 영속 컨텍스트 (`context/`) 운영 — 필수

> 채팅 기록은 휘발성입니다. `context/*.md` 파일만 영속 컨텍스트로 사용합니다.

### 사이클 시작 시 (필수 읽기)

작업을 시작하기 전에 필요한 최소 컨텍스트를 먼저 읽습니다. 모든 작업에 공통으로 아래 파일을 확인합니다:

1. `context/project_state.md` — 현재 프로젝트 상태
2. `context/failure_playbook.md` — 반복 실패 패턴/재발 방지 규칙
3. `context/open_tasks.md` — 진행 중/대기 중 작업
4. `context/decisions.md` — 의사결정 이력
5. `context/session_summary.md` — 최근 세션 요약

작업 유형에 따라 아래 문서를 추가로 읽습니다:

- Salesforce 설계/구현: `docs/architecture.md`, `docs/coding-standards.md`
- 배포/검증: `docs/deploy-gate-가이드.md`
- 오케스트레이션 변경: `docs/orchestration-가이드.md`
- 디자인/LWC 화면 작업: `docs/design/**`

### 사이클 종료 시 (필수 갱신)

의미 있는 상태 변화(요구사항 확정, 설계 결정, 구현/검증 결과, 신규 리스크)가 생기면 아래 파일을 갱신합니다:

1. `context/project_state.md` — 현재 목표, 구현 상태, 변경 파일, 테스트, 남은 작업, 리스크
2. `context/session_summary.md` — 이번 세션에서 한 일 요약
3. `context/decisions.md` — 의미 있는 설계 의사결정이 있을 때만
4. `context/open_tasks.md` — 신규 작업 추가, 완료 작업 제거
5. `context/failure_playbook.md` — 반복될 실패 패턴이 확인된 경우만 추가/갱신

### 압축 규칙

- 요약은 짧고 중복 없이 작성합니다.
- 오래된 진술은 제거하고 모순은 해소합니다.
- 기록 항목: 현재 목표, 구현 상태, 변경 파일, 테스트, 남은 작업, 리스크
- 큰 로그나 전체 diff는 저장하지 않습니다.

## 실패 기록 운영 (필수)

- 실패 로그는 `logs/failures/YYYY-MM/failure_YYYY-MM-DD.md`에 기록합니다.
- 기록 대상: 잘못된 구현 방향, 아키텍처 규칙 위반, 반복 가능성이 높은 구현 실수
- 로그 작성 후 반복 패턴이면 `context/failure_playbook.md`에 규칙 형태(`FP-XX`)로 반영합니다.
- `context`는 항상 작게 유지하고 최신 상태만 남깁니다. 상세 기록은 `logs`에 보관합니다.

## 재귀 개선 루프 (필수)

> 반복되는 문제와 반복되는 사용자 요청은 다음 작업 기준이 되도록 문서에 반영합니다.

### 개선 흐름

```text
발견 → 기록 → 승격 → 적용 → 검증
```

### 문서화가 필요한 상황

1. 같은 유형의 실수나 실패가 반복됨
2. 사용자가 같은 요청이나 선호를 여러 번 말함
3. AI가 매번 같은 확인 질문을 해야 함
4. Deploy Gate 또는 Review Gate에서 같은 유형의 항목이 반복됨
5. 기술부채 후보가 특정 패턴으로 계속 쌓임

### 반영 규칙

- 작업 방식 규칙은 `AGENTS.md`, `CLAUDE.md`에 반영합니다.
- 사람 기준 사용법은 `README.md`에 반영합니다.
- 오케스트레이션 흐름은 `docs/orchestration-가이드.md`에 반영합니다.
- 배포/검증 기준은 `docs/deploy-gate-가이드.md`에 반영합니다.
- 설계/코딩 기준은 `docs/architecture.md`, `docs/coding-standards.md`에 반영합니다.
- 반복 실패는 `context/failure_playbook.md`에 반영합니다.
- 자세한 기준은 `docs/recursive-improvement-가이드.md`를 따릅니다.

## 오케스트레이션 순서

```text
요구사항 명확화 → 기술설계 → 작업 snapshot(Salesforce 소스/Org 영향 시) → 디자인(필요시) → 구현 → 자체리뷰 → 변경/기술부채 보고 → 사용자 승인 → Org 반영
```

핸드오프 규칙:

- 다음 단계는 이전 단계 필수 산출물이 준비된 경우에만 시작합니다.
- 단계 생략/되돌림은 사유를 `context/decisions.md`에 기록합니다.

## 역할별 지침

### 요구사항 분석

- 요구사항 해석, 범위(in/out), 핵심 리스크를 명확히 합니다.
- 누락 정보가 결과나 리스크를 실질적으로 바꾸는 경우에만 사용자에게 질문하여 보완합니다.
- 확정된 요구사항은 `context/open_tasks.md`에 기록합니다.

### 기술설계

- 기술설계와 구현을 분리해 기록합니다.
- 변경 영향, 테스트 전략, 롤백 가능성을 명시합니다.
- 설계 의사결정은 `context/decisions.md`에 기록합니다.

### 디자인

- 기본은 SLDS 표준을 참고해 설계합니다.
- SLDS로 구현이 어려운 경우에만 커스텀을 진행합니다.
- 컴포넌트 간 일관적인 디자인을 유지합니다.
- 산출물은 `docs/design/specs/**` 경로에 작성합니다.

### 구현

- Salesforce 구현 시 보안/품질 규칙(CRUD/FLS, Bulkify 등)을 준수합니다.
- Salesforce 소스 또는 Org 영향 작업 전에는 영향 파일을 정리한 뒤 `npm run work:snapshot -- --target-org <ORG_ALIAS> --label <작업명> --files <파일...>`를 실행합니다.
- 문서/설정만 수정하고 Org에 영향이 없는 작업에는 Org snapshot을 요구하지 않습니다.
- `backups/` 아래 로컬 백업과 Org snapshot은 참고/복구용입니다. AI는 생성 및 참조만 가능하며 수정/삭제하지 않습니다.
- 구현 후 자체리뷰를 수행합니다.
- 파일 검색은 가능하면 `rg`를 우선 사용하고, 사용할 수 없으면 PowerShell 검색으로 대체합니다.
- 수동 파일 편집은 작고 검토 가능한 단위로 수행합니다.

### 자체리뷰 체크리스트

- [ ] CRUD/FLS 준수 여부
- [ ] Bulkify 기준 준수 (Trigger/Service)
- [ ] 하드코딩된 ID/값 없음
- [ ] 테스트 코드 포함 여부
- [ ] 기존 코드와의 충돌 가능성

### QA

- 기본 검증은 요구사항 기준으로 수행합니다.
- 사용성 불편 요소가 있으면 요구사항 범위를 벗어나지 않는 개선안을 제안합니다.
- 결함은 기준(요구사항/디자인/코드)과 심각도를 구분해 기록합니다.

## 기본 실행/반영 원칙

- 별도의 명시 승인이 없으면 소스 수정 후에도 Org에 배포하지 않습니다.
- 구현 후 생성/수정/삭제 파일 목록, 작업 요약, 검증 결과, 기술부채 후보를 사용자에게 보고합니다.
- 사용자가 배포를 승인한 경우에만 Deploy Gate를 통과한 안전 배포 명령을 사용합니다.
- 배포 직전 Org-aware 검사는 작업 시작 Org snapshot, 현재 로컬, 배포 직전 Org 최신본을 3-way 비교합니다.
- 서로 다른 영역의 변경은 자동 병합할 수 있고, 같은 의도/같은 영역 충돌 또는 삭제/의도 불명 변경은 중단 후 사용자에게 보고합니다.

## 배포 게이트 (Deploy Gate) — Org 반영 전 필수

> **Org 반영 전 체크 항목을 꼼꼼히 확인하고, 문제가 있으면 사용자에게 즉시 알리고 배포하지 않습니다.**

### 실행 방법

```bash
# OS 공통 규칙 검증
npm run deploy-gate:check

# OS 공통 안전 배포
npm run deploy:safe -- --target-org <ORG_ALIAS>
```

Windows 전용 래퍼:

```powershell
powershell -ExecutionPolicy Bypass -File scripts/deploy-gate-check.ps1
powershell -ExecutionPolicy Bypass -File scripts/deploy-with-gate.ps1 -TargetOrg <ORG_ALIAS>
```

macOS/Linux 전용 래퍼:

```bash
bash scripts/deploy-gate-check.sh
bash scripts/deploy-with-gate.sh <ORG_ALIAS>
```

### 검사 항목

**정적 검사**

1. 필수 문서/스크립트 파일 존재 여부
2. 금지 패턴(예: 외부 디자인 툴 참조) 검사
3. Markdown 로컬 링크 무결성
4. Markdown 파일 인코딩(UTF-8) 형식 검사
5. 디자인 인덱스 문서 내 경로 실존 여부

**배포 직전 Org-aware 검사**

1. 배포 대상 텍스트 소스가 UTF-8로 읽히는지 확인
2. 배포 대상 한글 텍스트 안에 깨짐 흔적이 끼어 있지 않은지 확인
3. 작업 시작 Org snapshot, 현재 로컬, 배포 직전 Org 최신본을 3-way 비교
4. 로컬에서 안 바꾼 파일인데 Org만 바뀐 경우 Org 최신본을 로컬에 반영
5. 서로 다른 메소드/라인을 바꾼 경우 자동 병합
6. 같은 메소드/라인 또는 삭제/의도 불명 변경은 **배포 중단**

**Review Gate**

1. 작업 snapshot의 영향 파일만 기술부채 후보 스캔
2. `TODO`, `FIXME`, `pending-confirm`, 임시/우회/가정, 하드코딩된 Salesforce ID 의심을 `review-needed`로 누적
3. Review Gate 항목은 배포를 자동 차단하지 않지만, 배포 승인 요청 전에 사용자에게 반드시 보고

### 실패 시 동작

- 위반 항목을 규칙 단위로 출력합니다.
- 종료 코드 `2`로 즉시 종료합니다.
- **배포 명령은 실행하지 않습니다.**
- **사용자에게 위반 내용을 즉시 보고하고 수정 방향을 제안합니다.**
