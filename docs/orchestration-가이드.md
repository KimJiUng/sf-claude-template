# 오케스트레이션 가이드

## 목적

Claude와 Codex 같은 AI 에이전트가 요구사항 수신부터 Org 반영까지 일관된 절차를 따르도록 합니다.

## 오케스트레이션 흐름

```text
요구사항 명확화 → 기술설계 → 작업 snapshot(Salesforce 소스/Org 영향 시) → 디자인(필요시) → 구현 → 자체리뷰 → 변경/기술부채 보고 → 사용자 승인 → Org 반영
```

## 단계별 상세

### 1. 요구사항 명확화

- 사용자의 요구사항을 목표, 성공 기준, 제약, 출력, 중단 조건으로 정리합니다.
- 누락 항목이 결과/보안/권한/데이터/배포에 영향을 주면 구현 전에 가장 작은 질문만 합니다.
- 저위험/가역 작업에서 합리적 가정이 가능하면 진행하고 가정과 확인 필요 항목을 보고합니다.
- 확정된 요구사항 또는 명시적 가정을 `context/open_tasks.md`에 기록합니다.

하네스 체크리스트:

1. 비즈니스 목적이 명확한가?
2. 대상 오브젝트/필드가 특정되었는가?
3. UI 요구사항이 있는 경우 레이아웃/동작 기준이 있는가?
4. 비즈니스 규칙(조건/예외/권한)이 정의되었는가?
5. 완료 조건(인수 기준)을 판단할 수 있는가?

### 2. 기술설계

- 변경 영향 분석을 수행합니다.
- 설계 의사결정을 `context/decisions.md`에 기록합니다.
- 테스트 전략과 롤백 가능성을 명시합니다.

### 3. 디자인 (필요시)

- SLDS 표준을 기본으로 사용합니다.
- 산출물은 `docs/design/specs/`에 작성합니다.

### 4. 작업 Snapshot

- Salesforce 소스 또는 Org에 영향을 주는 작업이면 영향 파일 목록을 확정합니다.
- 대상 Org alias가 확인되면 `npm run work:snapshot -- --target-org <ORG_ALIAS> --label <작업명> --files <파일...>`를 실행합니다.
- `backups/`에는 작업 전 로컬 백업과 작업 시작 시점의 Org snapshot을 저장합니다.
- 문서/설정만 수정하고 Org에 영향이 없는 작업에는 Org snapshot을 요구하지 않습니다.
- AI는 백업 파일을 생성/참조만 할 수 있고 수정/삭제하지 않습니다.

### 5. 구현

- `docs/architecture.md`의 계층 원칙을 준수합니다.
- 보안/품질 규칙(CRUD/FLS, Bulkify)을 준수합니다.

### 6. 자체리뷰

- CRUD/FLS 준수 여부
- Bulkify 기준 준수
- 하드코딩된 ID/값 없음
- 테스트 코드 포함 여부
- 기존 코드와의 충돌 가능성

### 7. 변경/기술부채 보고

- `npm run debt:scan` 또는 `npm run deploy-gate:check`로 Review Gate를 실행합니다.
- 생성/수정/삭제 파일 목록과 작업 내용을 요약합니다.
- `docs/technical-debt/register.md`에 누적된 `review-needed` 항목을 사용자에게 보여줍니다.

### 8. 사용자 승인

- 사용자가 명시적으로 승인하기 전에는 Org에 배포하지 않습니다.
- 승인 전 보고에는 작업 요약, 파일 목록, 테스트/검증 결과, 기술부채 후보, 자동 병합 여부를 포함합니다.

### 9. Org 반영

- Deploy Gate를 실행합니다.
- 위반이 있으면 **배포하지 않고** 사용자에게 보고합니다.
- Org 변경과 로컬 변경이 서로 다른 영역이면 자동 병합합니다.
- 같은 영역/같은 의도 충돌이나 삭제/의도 불명 변경은 배포를 중단하고 사용자에게 보고합니다.
- 위반이 없고 사용자가 승인한 경우에만 배포를 진행합니다.

## 단계 전환 규칙

- 다음 단계는 이전 단계 필수 산출물이 준비된 경우에만 시작합니다.
- 단계 생략/되돌림은 사유를 `context/decisions.md`에 기록합니다.
